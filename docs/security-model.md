# Security model

This document describes how `tiktok-mcp` handles secrets, log redaction, personally identifiable information, and write gating in v0.1. It enumerates the ten defense layers that ship with the server, the threats they address, and the matrix that maps the two together. It also states the limits of what an MCP server can promise and how to report a security issue.

If you find behaviour that contradicts what is written here, please report it via the channel in "Reporting security issues" at the end.

## Threat model

These are the adversaries and accidents we design against. Each one maps to one or more defense layers via the matrix later in this document.

1. **Malicious agent prompt-injects writes.** A user pastes a CSV, transcript, or web page into Claude. The pasted content contains a hidden instruction such as "pause campaign 12345". The agent reasons over that untrusted text and tries to call `pause_campaign` or `delete_ad`.
2. **Malicious account pastes phishing redirect URL.** During `complete_account_login`, the user pastes a `redirect_url` that points at an attacker-controlled domain or carries a forged `state` token. The MCP must refuse to exchange that code for tokens.
3. **Compromised app credentials.** A `client_secret` for one TikTok app leaks (committed to a public repo, posted in chat, intercepted on an unsecured network). The attacker wants to discover refresh tokens, swap accounts, or escalate to TikTok-side mutations.
4. **Lost laptop with unlocked keychain.** A developer's machine is stolen while their macOS Keychain is logged in. The attacker has shell access until the user remote-wipes or rotates credentials.
5. **Local log files leaked.** Logs from a debugging session land in a bug report, a Sentry issue, a Slack thread, or a vendor support ticket.
6. **Cassette tapes pushed with secrets or PII.** A contributor records a new `vcrpy` cassette and forgets to scrub it before committing.
7. **Weak AES key derivation on the disk fallback.** An attacker who reads the encrypted-file backend (`tokens.json.enc`) and the fernet key file mounts an offline brute-force attack.
8. **OAuth state replay.** An attacker captures a valid `state` token from a redirect URL and tries to reuse it to swap tokens for a different account.
9. **Refresh-token theft.** An attacker obtains a stored `refresh_token` and tries to mint access tokens long after the developer noticed the theft.
10. **PII in comment text.** Comment bodies from TikTok contain personal data (names, phone numbers, opinions). They land in logs, cassette tapes, or analytics dashboards by accident.

The "Known limitations" section lists what we explicitly do not protect against.

## Defense layers

### Layer 1: OS keychain primary storage

Tokens and app credentials are stored in the operating-system keychain via the `keyring` library: macOS Keychain, Windows Credential Manager, or Linux Secret Service (`libsecret`). The service name is `tiktok-mcp`. Each account record is serialised as JSON and indexed by a key like `tiktok-mcp::display::production::account::no-display-a4f1`. The keychain is the first thing we ask. We only fall back to disk (Layer 2) when `keyring` raises `NoKeyringError`, meaning there is no usable backend on the machine.

### Layer 2: AES-fernet encrypted file fallback

When no OS keychain is available (typical for headless Linux servers without a desktop session), `tiktok-mcp` writes to a single encrypted file at `platformdirs.user_data_dir("tiktok-mcp")/tokens.json.enc`. The cipher is `cryptography.fernet`, which is AES-128-CBC for confidentiality plus HMAC-SHA256 for integrity, both keyed off a 256-bit key generated by `Fernet.generate_key()`. The key itself is stored in the keychain when one is partially available, or in a sibling `fernet.key` file next to the encrypted store. Bootstrap with no key store at all is refused with a clear error. Plain unencrypted files are never an acceptable backend.

### Layer 3: SecretRedactor logging filter

`SecretRedactor` is a `logging.Filter` installed once on the root logger via `install_redactor()`. It transforms every log record before formatting in two ways:

1. **Pattern matching.** Seeded patterns redact `access_token=`, `refresh_token=`, `code=`, `client_secret=`, `auth_code=`, `secret=`, `Authorization: Bearer …`, `Access-Token: …`, and JSON-shaped `"access_token": "..."`. Matches are replaced with `<REDACTED>`.
2. **Runtime token registry.** Every token read from or written to the keychain (`serialize_account_record`, `deserialize_account_record`) calls `register_token(token, name)`. Subsequent log lines that contain that exact token are rewritten to `<REDACTED:access_token>` or `<REDACTED:refresh_token>`. Tokens are unregistered when the corresponding account is removed.

The filter is idempotent: calling `install_redactor()` twice does not double-filter records. There is no configuration knob to turn it off.

### Layer 4: httpx exception body sanitizer

TikTok 4xx and 5xx responses can contain copies of the request body, partial credentials, audience hash sources, or comment text. The default `httpx.Response.raise_for_status()` would propagate that body into traceback strings, where it would land in Sentry, logs, or stdout. We replace it with `safe_raise_for_status`, installed as an httpx event hook by `install_httpx_sanitization(client)`. The hook raises `SanitizedHttpxError` carrying only `status`, `url_path`, and the `request_id` extracted from the `x-tt-logid` response header. The original response body is discarded before the exception ever leaves the function.

### Layer 5: destructiveHint annotations and Claude Desktop permission prompts

Every write tool is annotated `ToolAnnotations(destructiveHint=True)`. Claude Desktop reads this annotation and prompts the user for permission on every invocation. The annotation is also enforced by the lint check in `assert_tool_decoration_compliance`: a write tool that forgets `@require_writes_enabled` or skips `destructiveHint` fails the test suite before it can ship. The user is therefore the final gate on every destructive call, even after they have set environment variables to permit writes server-side.

### Layer 6: TIKTOK_MCP_ALLOW_WRITES environment variable gate

The `@require_writes_enabled("<api>")` decorator wraps every write tool. At every call, the wrapper re-reads `os.environ.get("TIKTOK_MCP_ALLOW_WRITES")` and consults `parse_writes_env`. Re-evaluation per call means the user can toggle writes mid-session without restarting the MCP. When the gate blocks a call, the tool returns a structured error response with `error: "writes_disabled"`, the offending tool name, the API namespace, and a `would_have_done` hint. No partial mutation happens before the gate; the wrapped function is never entered.

### Layer 7: Per-API granularity of TIKTOK_MCP_ALLOW_WRITES

`TIKTOK_MCP_ALLOW_WRITES` accepts comma-separated tokens: `marketing`, `comments`, `posting`, `display`, or `all`. A advanced user can enable comment moderation while keeping ad-campaign mutations blocked with `TIKTOK_MCP_ALLOW_WRITES=comments`. Unknown tokens are logged at WARNING and ignored, so a typo such as `TIKTOK_MCP_ALLOW_WRITES=marekting` fails closed: nothing enables, the warning surfaces in stderr, and the operator sees that the intended namespace is still blocked.

### Layer 8: TIKTOK_MCP_ALLOW_LIVE_WRITES test-time gate

Tests tagged `@pytest.mark.live_write` are auto-skipped unless `TIKTOK_MCP_ALLOW_LIVE_WRITES=1` is set in the local shell. Continuous integration never sets this variable. The result: CI runs every write test under `vcrpy` replay only; no test can post a real campaign, comment, or video unless a developer explicitly opts in on their own machine. The split between `TIKTOK_MCP_ALLOW_WRITES` (production runtime gate) and `TIKTOK_MCP_ALLOW_LIVE_WRITES` (test-time gate) means CI cannot accidentally inherit a developer's local permission state.

### Layer 9: Two-step remove_account with confirmation_token

`remove_account(alias)` returns `{pending_removal: True, confirmation_token, expires_in: 60}` on the first call. The token is stored in an in-process map keyed by alias, with a 60-second TTL. The second call must pass `confirmation_token` back, and we compare it with `secrets.compare_digest` to prevent timing attacks. Expired or unknown tokens return `confirmation_expired_or_missing`; mismatched tokens return `confirmation_token_mismatch`. A single prompt-injected "delete that account" instruction will not complete the round trip on its own, because the model would need to observe the first response and reissue the call with the token within the TTL. The user sees the destructiveHint prompt twice (once per call).

### Layer 10: TIKTOK_MCP_ALLOW_ACCOUNT_CHANGES separate gate

`add_account`, `complete_account_login`, `remove_account`, `rename_account`, and `set_app_credentials` are wrapped with `@require_account_changes_enabled` rather than `@require_writes_enabled`. The orthogonal environment variable means a user onboarding their first account flips `TIKTOK_MCP_ALLOW_ACCOUNT_CHANGES=1` once, then sets it back to `0` or unsets it. From that point on, no further account add or remove succeeds even if `TIKTOK_MCP_ALLOW_WRITES=all` is set. This separates "Claude may mutate TikTok-side state" from "Claude may modify the local account inventory", which are different trust decisions.

## Threat-to-defense matrix

Every row below names a threat from the threat model and the defense layers that address it. The validator script in `tests/docs/validate_security_matrix.py` parses this table and asserts that every `Layer N` reference resolves to a heading above.

| Threat | Defenses |
|--------|----------|
| Prompt-injected campaign pause or ad delete | Layer 5, Layer 6, Layer 7 |
| Prompt-injected comment deletion or reply | Layer 5, Layer 6, Layer 7 |
| Prompt-injected account removal | Layer 5, Layer 9, Layer 10 |
| Prompt-injected app credential rotation | Layer 5, Layer 10 |
| Phishing redirect URL pasted into `complete_account_login` | Layer 3 redacts auth code in logs; Layer 5 still gates any resulting write; Layer 6 blocks mutations if tokens were minted |
| Compromised app credentials (`client_secret` leak) | Layer 1, Layer 2, Layer 3, Layer 10 |
| Lost laptop with unlocked keychain | Layer 6, Layer 7, Layer 10 narrow the blast radius until the user revokes |
| Local log file leaked into ticket or chat | Layer 3, Layer 4 |
| Cassette tape committed with raw tokens or PII | Layer 3 (same patterns used in cassette scrub), Layer 4, Layer 8 |
| AES fallback key recovered from disk | Layer 1 is preferred; Layer 2 isolates the fernet key under the user's data directory |
| OAuth state replay on a stale redirect URL | Layer 3 redacts code and state in logs; primary control is the single-use, ten-minute TTL state map outside the ten layers |
| Refresh-token theft from disk or memory | Layer 1, Layer 3, Layer 6, Layer 7 |
| Comment-body PII leaking into INFO logs | Layer 3 (token redaction), Layer 4 (HTTP body stripping); the comment-text rules in the PII section below extend this |
| Sandbox tokens leaking into production calls | Layer 1 namespacing (`tiktok-mcp::<api>::sandbox::…` vs `production::…`) plus Layer 6, Layer 7 |

## PII handling

The MCP touches three kinds of personal data: comment text, audience-upload hash sources (emails and phone numbers used for custom audiences), and TikTok user identifiers in API responses.

**Comment text.** Comment bodies from `list_comments`, `list_replies`, and related read tools are never cached to disk, never persisted to the keychain, and never logged at INFO. Cassette tapes under `tests/cassettes/**/*.yaml` are scrubbed before commit: every `text:` field longer than 50 characters is replaced with `[SCRUBBED]`. To dump raw comment bodies during debugging, two environment variables must be set together: `TIKTOK_MCP_LOG_LEVEL=DEBUG` (enable debug logging at all) and `TIKTOK_MCP_LOG_COMMENT_BODIES=1` (opt in to body content). Either one alone leaves bodies redacted. Production never sets either.

**Audience uploads.** When the MCP uploads a custom audience to Marketing API, the input file lists emails and phone numbers. We hash each row with SHA-256 over the trimmed, lower-cased string before transmission. The plaintext hash sources are not persisted by the MCP; the input file paths are not logged at INFO. The hashed output goes to TikTok over TLS.

**User identifiers.** Display API user IDs, advertiser IDs, and similar identifiers are not treated as secrets but are not logged at INFO unless they appear in a tool result. They are not redacted by Layer 3, by design: redacting all numeric IDs would make the logs useless. Operators concerned about ID leakage should use `TIKTOK_MCP_LOG_LEVEL=WARNING` in production deployments.

## Sandbox isolation

TikTok sandbox accounts and production accounts share neither tokens nor app credentials. Layer 1 keys carry the mode explicitly: `tiktok-mcp::marketing::sandbox::account::demo` versus `tiktok-mcp::marketing::production::account::demo`. Account records also carry a `sandbox: true | false` flag, and cross-mode access is rejected at the tool layer. A sandbox token leak therefore will not replay against production data, and a production token will not accidentally be sent to the sandbox endpoint by an agent that confuses the two.

## Known limitations

There are several things this server does not protect against, and the user is the final gate in each case.

1. **Prompt injection in the LLM itself.** If Claude (or any other agent) decides to call a write tool because of an injected instruction in pasted content, Layer 5 will surface a Claude Desktop permission prompt. If the user clicks Allow, the call proceeds. We do not detect or block adversarial pasted text inside the agent. The destructiveHint prompt, the per-API write gate, and the separate account-changes gate together narrow the blast radius, but they rely on a human noticing the prompt before approving.
2. **Compromised LLM client.** If the user is running a modified or malicious MCP client, all gates above are bypassed at the client edge. We assume an honest agent runtime.
3. **Endpoint compromise.** A laptop with full disk access by an attacker can read the encrypted fallback file plus its fernet key. Layer 2 protects only the cold-storage case; once an attacker has live process access, runtime tokens are reachable in memory.
4. **Network-level adversaries.** All TikTok traffic uses TLS, but we trust the operating-system certificate store. A user who installs a hostile root certificate is outside the model.
5. **TikTok-side abuses.** Token theft from TikTok's own infrastructure, or abuse by TikTok staff, is outside the scope of an MCP server.

No absolute claim is made here. The defenses are a layered system intended to make accidents and casual abuse expensive, not to stop a targeted, well-resourced adversary with endpoint access.

## Reporting security issues

Please report security issues privately rather than in a public GitHub issue:

- Open a [GitHub Security Advisory](https://github.com/Signikant/tiktok-mcp/security/advisories) on the repository, or
- Email `[email protected]`.

Please include a description of the issue, reproduction steps, and the version of `tiktok-mcp` (output of `tiktok-mcp --version` once that flag ships in v0.1, or the installed PyPI version). We aim to acknowledge reports within five business days and to publish a patched release within thirty days of confirmation.
